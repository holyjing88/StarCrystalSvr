## 3. IDIP 路径与运营登录

### 3.0 IDIP 路径命名（与玩家 API 区分）

**根前缀区分**（资源段可与玩家同名，靠第一层路径识别）：

```text
玩家（Unity / H5 业务）     /api/v1/{资源}/...
运营 IDIP                  /idip/v1/{资源}/...
```

| 玩家 API（示例） | IDIP（本文档 · 新） | 说明 |
|------------------|---------------------|------|
| `POST /api/v1/auth/guest` | `POST /idip/v1/auth/login` | 运营登录，非玩家鉴权 |
| — | `POST /idip/v1/auth/logout` | |
| — | `POST /idip/v1/auth/heartbeat` | |
| `GET /api/v1/games` | `GET /idip/v1/games/list` | 全量含 offline |
| — | `POST /idip/v1/games/upsert` | |
| — | `POST /idip/v1/games/batch-upsert` | |
| — | `POST /idip/v1/games/delete` | 仅 offline；可选删 H5 目录（§5.4） |
| — | `POST /idip/v1/games/h5/upload` | |
| — | `POST /idip/v1/publish/rsync-retry` | |
| — | `GET /idip/v1/audit/logs` | |

**存量 IDIP**（同规则，已实现）

| IDIP 路径 | 说明 |
|-----------|------|
| `/idip/v1/gold/*` | 金币运营 |
| `/idip/v1/welfare/*` | 福利结算 |
| `/idip/v1/tasks/*` | 任务配置（玩家侧为 `/api/v1/tasks/*`） |

**请求头 / 响应**：`X-IDIP-Session`、`X-IDIP-Key`；JSON 信封 `{ code, message, data }`（与 `server-go/doc/IDIP_API.md` 一致）。

**实现**：`idip-webclient` 调用上表路径；Vite 代理 `/idip` → server-go。

---

### 3.1 运营登录

```text
内网 IP
  POST /idip/v1/auth/login       → username + password（无需 Session/Key）
  POST /idip/v1/auth/heartbeat   → X-IDIP-Session（§3.7）
  其它 /idip/v1/*                     → 满足其一：
        ① 有效 X-IDIP-Session（且心跳未超时）
        ② 有效 X-IDIP-Key（与 starcrystal.json idip.key 一致，供测试/脚本）
```

- **idip-webclient**：登录后用 **Session + 心跳**；UI **不展示** Key。
- **Vitest / 旧脚本**：可继续只带 **`X-IDIP-Key`**，无需登录。
- 二者同时提供时：**Session 优先**；Session 无效时再校验 Key。

### 3.2 单端登录

Redis 维护：

```text
idip:session:{token}           → { username, expiresAt }
idip:user-active:{username}    → {token}    # 单端：新 login 删除旧 token 键
```

同一账号在 B 处登录后，A 处旧 `X-IDIP-Session` 立即 **401**。

### 3.3 配置示例

`release/configs/starcrystal.json`：

```json
{
  "idip": {
    "operatorCipherKey": "<base64-32-bytes>",
    "operators": [
      { "username": "ops_admin", "passwordEnc": "v1:..." }
    ],
    "sessionTtlSec": 28800,
    "sessionHeartbeatIntervalSec": 30,
    "sessionHeartbeatTimeoutSec": 30,
    "loginFailMaxAttempts": 5,
    "loginFailWindowSec": 60
  },
  "publish": {
    "h5AssetsDir": "",
    "gamesConfigPath": "",
    "h5BackupDir": "",
    "gamesJsonBackupMax": 10,
    "rsync": {
      "enabled": false,
      "targets": []
    }
  }
}
```

| 字段 | 缺省（空字符串） |
|------|------------------|
| `publish.h5AssetsDir` | `{repoRoot}/release_h5` |
| `publish.gamesConfigPath` | `{repoRoot}/release/configs/games.json`（`GAMES_CONFIG` 环境变量若已设则优先） |
| `publish.h5BackupDir` | `{repoRoot}/release_h5_backup` |
| `publish.gamesJsonBackupMax` | `10` |
| `idip.loginFailMaxAttempts` | `5`（见 §3.6） |
| `idip.loginFailWindowSec` | `60` |
| `idip.sessionHeartbeatIntervalSec` | `30` |
| `idip.sessionHeartbeatTimeoutSec` | `30`（与间隔相同，丢 1 次心跳即 401） |

`cwd` = server-go 二进制启动时的 **当前工作目录**（与现网 `release/startsvr.sh` 一致）。

**生成 `passwordEnc`**

| 脚本 | 说明 |
|------|------|
| `tools/idip-webclient/scripts_encrypt/encrypt-idip-operator.sh` | Linux：**无参数**；默认 `ops_admin`；自动写入 `release/configs/starcrystal.json` |
| `tools/idip-webclient/scripts_encrypt/encrypt-idip-operator.ps1` | Windows：`-Username` / `-Password` / `-CipherKeyBase64` 可选；默认写配置 |

底层：`go run ./cmd/encrypt-idip-operator`（AES-256-GCM，`v1:` + base64）。`encrypt-idip-operator.sh` / `.ps1` 在未设置 `IDIP_OPERATOR_CIPHER_KEY` 时会自动生成 32 字节随机密钥（Base64，含字母/数字/符号）；生产请保存该值到 `operatorCipherKey` 或 env。

### 3.4 idip-webclient

- 登录页：账号 + 密码 + 可选 API Base
- 成功：`sessionStorage` 存 `idip.sessionToken`；启动 **心跳定时器**（§3.7）
- **无** `X-IDIP-Key` 配置栏
- 退出 / 401 → 清 Session → 回登录页

### 3.5 `auth` 接口契约（P0）

**`POST /idip/v1/auth/login`**

```json
// 请求
{ "username": "ops_admin", "password": "…" }

// 成功 data
{ "sessionToken": "…", "expiresAt": "2026-05-29T20:00:00+08:00" }
```

- 响应头可回显 `X-IDIP-Session: {sessionToken}`（与请求头同名，客户端二选一存储即可）
- 失败：用户名/密码错误 → **401**，`code=1401`；限流 → **429**，`code=1429`（§3.6）

**`POST /idip/v1/auth/logout`**

- 请求头：`X-IDIP-Session`
- 成功 `data`：`{}`；无效 Session 亦视为成功（幂等）

**`POST /idip/v1/auth/heartbeat`**

- 请求头：`X-IDIP-Session`；body 空
- 成功 `data`：`{ "expiresAt": "…" }`（绝对上限不变，仅刷新 `lastHeartbeatAt`）

### 3.6 登录失败限流

按 **用户名 + 来源 IP** 在 Redis 计数：

```text
idip:login-fail:{username}:{clientIp}  →  失败次数，TTL = loginFailWindowSec
```

| 项 | 默认值 |
|----|--------|
| `loginFailMaxAttempts` | 5 |
| `loginFailWindowSec` | 60 |

窗口内失败次数 ≥ 上限 → **429**，`code=1429`；成功登录后清除该计数。开发态来源 IP 为 `127.0.0.1` **接受**。

### 3.7 会话心跳

**原则**：通过 **心跳** 判断运营台会话是否仍在线；非每次业务请求滑动续期。

Redis Session 字段扩展：

```text
idip:session:{token}  →  { username, expiresAt, lastHeartbeatAt }
```

| 项 | 值 | 说明 |
|----|-----|------|
| `sessionHeartbeatIntervalSec` | 30 | 客户端每 30s `POST /idip/v1/auth/heartbeat` |
| `sessionHeartbeatTimeoutSec` | 30 | 超过 30s 无心跳 → **401** |
| `sessionTtlSec` | 28800 | 绝对上限，与心跳 **并行** |

```text
idip-web 主界面：每 30s POST /idip/v1/auth/heartbeat → 更新 lastHeartbeatAt

鉴权检查（任一满足即 401）：
  now > expiresAt（超过 8h 绝对上限）
  或 now - lastHeartbeatAt > sessionHeartbeatTimeoutSec

仅 X-IDIP-Key：不要求心跳
```

被新登录踢下线时：删旧 Session + 审计 **`login_kick`**（§6.5；P3 前审计 no-op，§8.1）。

## 4. H5 上传与发布

### 4.1 Zip 约定

```text
web-mobile-planefight.zip          ← 文件名（去 .zip）须等于顶层目录名
└── web-mobile-planefight/
    ├── index.html
    └── ...
```

校验：`basename(file, ".zip")` **==** 唯一顶层目录名（§2.5 #55）。

解压结果：`{h5AssetsDir}/web-mobile-planefight/index.html`

### 4.2 `gameId` 与 `gameDirName`

| 字段 | 来源 |
|------|------|
| `gameId` | **idip-web 表单**（如 `g005`） |
| `gameDirName` | zip **唯一顶层目录名**（如 `web-mobile-planefight`） |

服务端生成：

```text
entryUrl = h5/{gameDirName}/index.html?v={minigameVersion}
downloadUrl = h5/{gameDirName}_v{minigameVersion}.tar.gz
```

**绑定规则**（`gameDirName` 自 `entryUrl` 路径解析，如 `h5/{gameDirName}/index.html`）：

> **客户端**：`entryUrl` / `downloadUrl` 为相对路径，由 `gameBaseCDNUrl-*` 拼接为绝对 URL（见 Unity 工程 `doc/H5小游戏CDN路径与gameBaseCDNUrl设计.md`）。  
> **服务端磁盘**：H5 仍解压到 `{repoRoot}/release_h5/`（`publish.h5AssetsDir`）；HTTP 对外路径为 `/h5/*`（与 CDN `h5/...` 一致）。

| 场景 | 行为 |
|------|------|
| **create**：`gameId` 不在 json | 解压成功后 **新增** json 条目 |
| **update**：`gameId` 在 json 且 zip 目录名与 entryUrl **一致** | 备份 → 覆盖解压 → 更新 json |
| **create/update**：目录已被 **其它 `gameId`** 占用 | **400** |
| **create/update**：同 `gameId` 但 zip 目录名与 json **不一致** | **400** |
| **`status=offline`** 同 `gameId`+同目录 upload | **允许**（§2.5 #57）；客户端列表仍不展示 offline |

### 4.3 `minigameVersion`（四段版本）

- 格式：`^\d+\.\d+\.\d+\.\d+$`
- **强制**：`games.json` 中每条 `entryUrl` **必须**含合法四段 `v=`（§2.5 #53）
- **upsert**：可带 `minigameVersion` 字段但 **忽略**（不改 `v=`）；仅 **upload** 可升版本
- 上线前：运维脚本 **自动**补 `1.0.0.0`（§12.3）

**upload 自动分流（§2.6 #67）**

```text
gameId ∉ games.json                 → create（解压成功后新增条目）
gameId ∈ json 且 gameDirName 一致   → update（备份覆盖流）
否则                                → 400
```

- **upload** 时新 `minigameVersion` 须 **大于** 该 `gameId` 当前 `v=`

### 4.4 覆盖上传流程

前提：同 `gameId` + 同 `gameDirName`（自 entryUrl 解析）；**offline 不阻断** upload。

```text
0. 获取 games.json 全局写锁（§2.4 #42）
1. 校验 minigameVersion 四段递增
2. 解压到临时目录 → 校验最小文件集（**至少 index.html**；Cocos 包可含 main.js、src/settings.js）
3. 若目标目录已存在：整目录 zip 备份 → `h5BackupDir/{gameDirName}/`（§12.1）
4. 临时目录原子 rename 到 {h5AssetsDir}/{gameDirName}/
5. 备份 games.json → games.json.bak.{timestamp}（超 10 删最旧）
6. 更新 games.json（保持原文件排版格式 §2.4 #41）
7. 任一步失败 → 回滚（删新目录、恢复旧目录若已删、不写 json）
8. 写审计 `h5_upload`（§8.1）；释放写锁
9. 若启用 rsync → 全部 target 成功（§6.3）
```

**新增游戏**（目录不存在、json 无冲突）：跳过步骤 3 的备份删旧，其余相同。

**禁止**：未解压成功即写 json；**禁止**仅 json 占位 create（§2.4 #40）。

### 4.5 上传表单

| 字段 | 必填 | 说明 |
|------|------|------|
| `gameId` | 是 | 运营指定，与目录名无关 |
| `minigameVersion` | 是 | 四段版号，如 `1.1.1.0` |
| `name` / `nameEn` / `nameUr` | name 必填 | |
| `channels` / `iconLink` / `coverUrl` / `sort` / … | 见 v0.2 | |
| `entryType` | 是 | `h5` |
| `status` | 是 | `online` / `offline` |

**无** `targetSubDir`、**无** `mode`（自动 create/update）。

### 4.6 `POST /idip/v1/games/h5/upload`

`multipart/form-data`：`file` + `meta`（JSON）。

## 5. 游戏列表

### 5.1 `GET /idip/v1/games/list`

全量 `list`（含 offline）+ `configVersion` + `configPath`。  
每条若 `entryUrl` 缺/非法四段 `v=`：`missingVersion: true`（前端 **标红**，§2.6 #66）。

### 5.2 `POST /idip/v1/games/upsert`

- `gameId` **必填**且须 **已存在**；**不可**通过 upsert 新增条目（新增仅 **upload**）
- 带 `expectedConfigVersion`；冲突 → **409**
- 请求体 **可含** `minigameVersion` 字段，服务端 **忽略**（不改 `v=`）
- 可改 `name`、`status`、`channels`、`sort` 等；**不可**改 `gameDirName`
- 写盘前 `games.json.bak.{timestamp}`；`.bak.*` 最多 **10** 份；保持原排版

### 5.3 `POST /idip/v1/games/batch-upsert`

```json
{
  "expectedConfigVersion": "…",
  "items": [ { "gameId": "g001", "name": "…", "status": "online", … } ]
}
```

- 同一 `expectedConfigVersion` 下批量写盘；`items` **≤ 50**（§2.6 #69）
- 每条规则同 §5.2；**任一**条目非法 → **整批失败**（不写盘）
- 不涉及 `v=` 变更

### 5.4 `POST /idip/v1/games/delete`

删除 **`games.json` 中已有配置项**（非 upload 新建路径的逆操作）。

**请求体**

```json
{
  "gameId": "g005",
  "expectedConfigVersion": "…",
  "deleteH5Dir": false
}
```

| 字段 | 必填 | 说明 |
|------|------|------|
| `gameId` | 是 | 须已存在于 `list` |
| `expectedConfigVersion` | 是 | 与 §5.5 乐观锁一致；不一致 → **409** |
| `deleteH5Dir` | 否 | 默认 **`false`**；`true` 时额外删除 `{h5AssetsDir}/{gameDirName}/` |

**校验**

| 条件 | 结果 |
|------|------|
| `gameId` 不存在 | **400** |
| `status != offline` | **400**（须先 `upsert` 下线） |
| `expectedConfigVersion` 与现盘 SHA256 不一致 | **409** |

**流程**

```text
0. 获取 games.json 全局写锁（§2.4 #42）
1. 加载条目；校验 offline
2. 自 entryUrl 解析 gameDirName
3. 备份 games.json → games.json.bak.{timestamp}
4. 从 list 移除该 gameId；写盘（保持原排版 §2.4 #41）
5. 若 deleteH5Dir=true：递归删除 {h5AssetsDir}/{gameDirName}/
   → 失败：用步骤 3 备份回滚 json，**500**（不写审计成功）
6. 写审计 `games_delete`（§8.1）；释放写锁
7. 若启用 rsync → 同步 games.json（及 deleteH5Dir 时同步远端目录删除，§6.3）
```

**成功 `data`（示例）**

```json
{
  "gameId": "g005",
  "configVersion": "…",
  "gameDirName": "web-mobile-planefight",
  "h5DirDeleted": false
}
```

**运营台**：游戏列表对 `offline` 行提供「删除」；二次确认；勾选「同时删除 H5 目录」→ `deleteH5Dir=true`。

### 5.5 `configVersion`（SHA256）

**`configVersion` 不是 `starcrystal.json` 里的配置项。**

算法：**SHA256(`games.json` 文件原始字节)** → 小写十六进制字符串。

```text
读取 games.json
  → configVersion = hex(SHA256(fileBytes))
  → 随 GET /idip/v1/games/list、GET /api/v1/games 返回

运营台保存：
  → expectedConfigVersion = 上次拉取的 SHA256
  → 服务端现算 SHA256，不一致 → 409

写入成功后：
  → 返回新 SHA256 作为 configVersion
```

运营台在「拉取列表」时缓存 `configVersion`，「保存」时原样带回。

### 5.6 玩家 API `GET /api/v1/games` 与 `v=`

- 读盘后过滤 `status=online` 等既有逻辑
- 若某条 `entryUrl` **缺/非法四段 `v=`**：**`logger.Error` 记日志 + 跳过该条**（§2.6 #65），不拖垮整表

## 6. 审计日志（MySQL）

### 6.1 表结构（建议）

```sql
CREATE TABLE idip_audit_log (
  id          BIGINT UNSIGNED NOT NULL AUTO_INCREMENT PRIMARY KEY,
  created_at  DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3),
  username    VARCHAR(64) NOT NULL,
  action      VARCHAR(32) NOT NULL,
  game_id     VARCHAR(64) NULL,
  detail      JSON NULL,
  KEY idx_created (created_at),
  KEY idx_user_time (username, created_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

`action` 示例：`login`、`logout`、`login_kick`、`h5_upload`、`games_upsert`、`games_delete`、`rsync_failed`。

**不记录**：登录密码错误、429 限流（§2.5 #52）。

### 6.2 保留策略

- 保留 **30 天**
- 每日定时任务或写入时批量 `DELETE WHERE created_at < NOW() - INTERVAL 30 DAY`

### 6.3 多机 rsync（可选）

当 **`publish.rsync.enabled = true`**（**仅一台固定主写节点**，§2.5 #60）时，本机写盘成功后 **rsync** 到各 `targets[]`。

配置示例：

```json
{
  "publish": {
    "rsync": {
      "enabled": true,
      "targets": [
        {
          "host": "10.0.0.12",
          "user": "deploy",
          "remoteH5AssetsDir": "/app/starcrystal-release/release_h5",
          "remoteGamesConfigPath": "/app/starcrystal-release/release/configs/games.json",
          "remoteH5BackupDir": "/app/starcrystal-release/release_h5_backup",
          "sshPort": 22
        }
      ]
    }
  }
}
```

**同步范围**

| 事件 | rsync 内容 |
|------|------------|
| `h5/upload` 成功 | `{gameDirName}/` → `remoteH5AssetsDir/{gameDirName}/` |
| `games/delete` 且 `deleteH5Dir=true` | 远端 `ssh rm -rf remoteH5AssetsDir/{gameDirName}`（**不用**全量 rsync `--delete`） |
| 写 `games.json` | `games.json` → `remoteGamesConfigPath` |
| 备份产生时 | `games.json.bak.*` → 远端 `configs/` 同目录 |
| H5 目录备份 zip | `h5BackupDir/{gameDirName}/*.zip` → 远端 `remoteH5BackupDir/{gameDirName}/` |

**目录约定（§2.6 #70）**

```text
{repoRoot}/release_h5/              ← publish.h5AssetsDir 默认
{repoRoot}/release_h5_backup/       ← publish.h5BackupDir 默认
{cwd}/configs/games.json
```

未配置 `remoteH5BackupDir` 时，默认 `{repoRoot}/release_h5_backup`（与 `release_h5` 同级）。

**约定**

- 使用系统 `rsync`（**不带 `--delete`**）；SSH 密钥由运维预置。
- **全部 target 必须成功**；任一失败 → **500**，审计 `rsync_failed`；本机写盘不回滚。
- `enabled: false` 时仅写本机。

### 6.4 rsync 重试

`POST /idip/v1/publish/rsync-retry`

```json
{
  "kind": "h5_dir",
  "gameDirName": "web-mobile-planefight"
}
```

或

```json
{ "kind": "games_json" }
```

- 根据最近一次本机成功写盘记录（或请求体）对 **所有 target** 重新 rsync；仍须 **全部成功**。
- 鉴权：Session 或 Key。

### 6.5 审计（MySQL + 运营台只读）

**库表（G1）**：与 **现有业务库同 MySQL 实例**（`starcrystal.json` → `mysql` 连接），不单独建库。

```sql

  action      VARCHAR(32) NOT NULL,
  game_id     VARCHAR(64) NULL,
  detail      JSON NULL,
```
