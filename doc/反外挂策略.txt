使用 Android KeyStore 存储：
不要把生成的设备识别码简单存在文件里。存入硬件安全模块（KeyStore），这样即便用户卸载重装，甚至清理缓存，你的 APP 依然能从硬件中读回上一次生成的同一个 ID。

建立“设备-用户”映射表：
在 MySQL 中建立一张表，记录 device_id 和 user_id 的关联。

当新用户进入时，查询该 device_id 关联了多少个 user_id。

如果超过阈值（如 5 个），则提示“该设备注册频繁”，强制要求绑定手机号（解决你 Jazz 卡收短信问题后）或进行真人扫脸。

封禁策略：
当某个 user_id 被判定为外挂，连坐该 device_id 下的所有账号，并拒绝该 device_id 未来所有的游客登录请求。

总结：
如果你只看系统 API 返回的 ID，它在同一台设备上理应是不变的；但作为开发者，你必须主动采取技术手段（如 KeyStore）去“记住”这个 ID，否则外挂玩家有一百种方法让你的 APP 觉得他换了台新手机。


首登时间间隔：

如果一个设备 ID 在 10 分钟内连续注册了 5 个游客账号，这绝对是脚本，直接封。

同一个设备ID，一天时间内 如果超过5个账户登录，则启动反外挂策略，1天时间内 该设备下的所有账户都不给广告。

同一个设备ID，一天时间 刷出的广告数是一定的，每个账户 必须记录 当天已经刷了多少次广告数

广告点击率 (CTR) 异常：

正常玩家玩三消时，看广告的频率是有波动的。如果一个账号每隔精确的 31 秒就点一次广告（跳过广告后的最小间隔），大概率是模拟器挂机。

设备特征指纹：

检查系统 API 返回的手机型号（Model）、分辨率、系统版本。

报警信号： 如果 10 个账号都来自同一型号、同一系统版本，且传感器数据（陀螺仪、加速度计）完全静止，这是典型的云手机或群控设备。

钱包提现门槛：

允许一个设备 ID 挂多个号玩，但限制一个设备 ID 每天只能提现 X 次，或者提现必须绑定唯一手机号。这从经济利益上直接阻断了职业刷子的动力。
